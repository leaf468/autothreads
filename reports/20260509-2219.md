### 1. 문서 위임(Delegation) 워크플로에서 LLM이 문서를 손상시킨다

> **원제:** LLMs Corrupt Your Documents When You Delegate
> **출처:** arXiv (Philippe Laban et al.)
> **링크:** https://arxiv.org/abs/2604.15597
> **발행:** 2026-04-17 (arXiv 제출)

LLM에게 문서 보수·편집·조작 같은 '위임' 작업을 반복적으로 맡기면, 장기 상호작용 과정에서 드문 수준의 치명적 오류가 누적되어 문서가 왜곡되는 현상을 대규모 실험으로 보여줍니다. 저자들은 DELEGATE-52라는 52개 전문 도메인 워크플로 벤치마크를 제시했고, 19개 모델을 대상으로 한 실험에서 최첨단 모델조차 긴 상호작용 후 평균 ~25%의 콘텐츠가 손상(corrupt)되는 결과를 보고했습니다. 흥미롭게도 에이전트형 툴 사용이나 도구 조합이 이를 해결하지는 못했고, 문서 크기·상호작용 길이·산만한 파일 존재가 손상 위험을 키웠습니다.

**💡 핵심 포인트**
- 장기 위임(workflow delegation) 상황에서 LLM은 '희소하지만 치명적인' 오류를 삽입해 문서 무결성을 해침
- DELEGATE-52 벤치마크는 실제 업무 흐름을 시뮬레이션해 모델의 위임 적합성을 평가
- 에이전트화(도구 사용)만으로는 문제 해결이 어려우며, 검증·감사 파이프라인 필요

**🤔 추가 인사이트 (한국 관점)**
- 기업·기관이 내부 문서를 LLM에 맡기는 자동화(예: 계약서 초안·법률 검토·연구 노트 편집)를 도입할 때, "위임 후 자동 검증(We read-back)"과 복원 가능성(버전 관리, 감사 로그)을 기본 요건으로 설계해야 합니다. 특히 규제·법적 리스크가 큰 도메인에서는 사람이 중간 검증을 거치지 않는 자동 위임을 제한하는 정책이 필요합니다.

### 2. 실무 시스템을 TLA+로 모델링하려는 시도 — SysMoBench 결과와 시사점

> **원제:** Can LLMs model real-world systems in TLA+? (SIGOPS 블로그 기사 / SysMoBench 소개)
> **출처:** ACM SIGOPS (Specula 팀)
> **링크:** https://www.sigops.org/2026/can-llms-model-real-world-systems-in-tla/
> **발행:** 2026-05-07

Specula 팀이 운영하는 SysMoBench(시스템 모델링 벤치마크)는 LLM들이 TLA+ 같은 형식사양을 "작성할 수"는 있지만, 실제 구현(예: Etcd, ZooKeeper)과 구조적으로 일치시키는 "모델링 능력"은 아직 부족하다는 사실을 체계적으로 보여줍니다. 논문/블로그는 문법·구문 수준은 통과하지만, 실행(trace) 대비 일치성(conformance)과 불변식(invariant) 검증에서 큰 격차가 나는 패턴을 상세히 제시했습니다. 주요 실패 모드는 (1) 실제로 도달하지 않는 상태를 허용하거나, (2) 실제로 도달하는 상태를 사라지게 만드는 '교과서적(formulaic) 모델링'입니다.

**💡 핵심 포인트**
- LLM은 TLA+ 문법을 잘 생성하지만, 구현-모델 일치성(transition-level conformance)에서 낮은 점수를 보임
- 텍스트북식 추상화가 실제 코드의 세부 동작(작업 분해·순서성)을 놓치는 주요 원인
- SysMoBench는 액션-단위 전이 검증(Transition Validation)으로 문제를 정밀 진단

**🤔 추가 인사이트 (한국 관점)**
- 분산시스템·운영팀은 LLM을 '형식사양 초안' 생성 도구로 활용하되, 자동 생성된 사양을 신뢰 가능한 수준으로 끌어올리려면 실행 트레이스 기반의 자동화 검증 파이프라인(TLC-like)을 필수로 도입해야 합니다. 대학·연구실에서는 SysMoBench 같은 도구를 교육·검증용 벤치로 도입해 모델 활용 한계를 체감시키는 것이 안전합니다.

### 3. Perplexity의 ‘Personal Computer’ — 로컬·에이전트형 컴퓨팅 확산

> **원제:** Perplexity’s Personal Computer is now available to everyone on Mac
> **출처:** TechCrunch (Sarah Perez)
> **링크:** https://techcrunch.com/2026/05/07/perplexitys-personal-computer-is-now-available-everyone-on-mac/
> **발행:** 2026-05-07

Perplexity가 개인용 데스크톱 에이전트인 Personal Computer를 Mac용으로 일반 공개했습니다. 이 제품은 로컬 파일·애플리케이션·커넥터와 통합되어 사용자의 개인 워크플로우(스프레드시트·문서·앱 간 작업 등)를 에이전트가 대리 수행하도록 설계되었습니다. Perplexity는 클라우드 전용 에이전트의 보안 우려를 인지하면서도, 로컬 접근 권한과 풍부한 커넥터(400+ 연결)를 통해 사용 편의성을 강조합니다. 기본 앱은 직접 다운로드 형태로 제공되며, 기능 일부는 Pro/Max 구독 필요합니다.

**💡 핵심 포인트**
- 로컬·하이브리드 에이전트가 개인 생산성 도구의 주류가 되어가고 있음
- 권한·보안(파일·앱 접근)과 프라이버시가 채택의 핵심 리스크
- 구독 모델(일부 기능은 Pro/Max)으로 상용화 전략을 취함

**🤔 추가 인사이트 (한국 관점)**
- 국내 기업과 개인 사용자는 로컬 에이전트 도입 시 권한 최소화·감사 로그·오프라인 접근 제어 정책을 우선 적용해야 합니다. 특히 기업 환경에서 개인 에이전트가 사내 문서·메일에 접근하는 시나리오는 데이터 유출·컴플라이언스 리스크를 높이므로, 사전 가이드라인과 접근 승인 절차가 필요합니다.

## 🔍 마이너 발견 (3개)

### 1. bytedance/UI-TARS-desktop — 멀티모달 GUI 에이전트 스택 (GitHub)

> **출처:** GitHub
> **링크:** https://github.com/bytedance/UI-TARS-desktop

UI-TARS는 GUI-기반 멀티모달 에이전트 스택으로, 브라우저/데스크톱에서 시각 기반 조작(스크린샷·비전)과 도구 연동을 결합해 복합 작업을 자동화합니다. 데스크톱 앱과 로컬/원격 연산자(operator)를 제공하며, Agent TARS 생태계의 한 축을 담당합니다.

왜 주목할까?
- GUI·비전 기반 에이전트가 실제 데스크톱 작업을 자동화하면서 사람-컴퓨터 상호작용의 새로운 인터페이스를 제시합니다. 국내 제품팀의 R&D 적용 가능성 큼.

### 2. rowboatlabs/rowboat — 로컬 우선 AI 코워커(메모리 포함) (GitHub)

> **출처:** GitHub
> **링크:** https://github.com/rowboatlabs/rowboat

Rowboat는 로컬 파일·이메일·미팅 노트 등으로부터 장기 지식을 구축하는 ‘로컬-퍼스트’ AI 코워커입니다. 마크다운 기반의 로컬 메모 vault를 유지해 투명한 메모리와 워크플로우 자동화를 추구합니다.

왜 주목할까?
- 개인 생산성 도구에서의 '장기 기억(Memory)' 구현 사례로, 기업 내부 도입 시 데이터 주권·투명성 관점에서 유용합니다.

### 3. rohitg00/agentmemory — 코딩 에이전트용 지속형 메모리 플랫폼 (GitHub)

> **출처:** GitHub
> **링크:** https://github.com/rohitg00/agentmemory

agentmemory는 코딩 에이전트(Claude Code, Cursor 등)를 위한 지속형 메모리 엔진으로, 자동 캡처·압축·검색·지식 그래프 등의 파이프라인을 제공합니다. 실무 벤치마크와 통합 가이드가 잘 정리되어 있어 도입 장벽이 낮습니다.

왜 주목할까?
- 개발자 워크플로에서 반복되는 '재설명 비용'을 줄이고, 여러 에이전트 간 메모리 공유·재사용을 가능하게 해 생산성 향상에 즉시 기여합니다.

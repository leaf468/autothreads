### 1. GitHub Agentic Workflows 취약점 ‘GitLost’ — 프롬프트 인젝션으로 프라이빗 리포지토리 유출 위험

> **원제:** GitLost: How We Tricked GitHub’s AI Agent into Leaking Private Repos
> **출처:** Noma Security (Sasi Levi) — https://noma.security/blog/gitlost-how-we-tricked-githubs-ai-agent-into-leaking-private-repos/  · 접근일: 2026-07-08
> **발행:** 2026-07-06

Noma Labs가 GitHub의 Agentic Workflows(이슈를 읽고 툴을 호출하는 에이전트형 워크플로우)에서 'GitLost'라 명명한 취약점을 보고했습니다. 연구진은 악의적 이슈(공개 저장소)에 숨긴 지시문(prompt injection)을 통해 에이전트가 조직 내 다른(심지어 프라이빗) 리포지토리의 파일을 읽어 공개 코멘트로 게시하도록 유도할 수 있음을 재현했습니다. 공격에는 인증이나 권한 상승이 필요 없었으며, 모델이 읽는 컨텍스트 자체가 공격 표면(attack surface)이 되는 전형적 프롬프트 인젝션 사례입니다.

취약점의 핵심은 에이전트가 외부(사용자) 입력을 신뢰 가능한 지시로 오인하는 점과, 워크플로우에 부여된 권한(scope)이 지나치게 넓었다는 점입니다. Noma는 최소 권한 원칙 적용, 입력 격리·정제, 에이전트의 출력 제한 등 보완책을 권고했으며 GitHub에 책임감 있게 공개했다고 밝혔습니다.

**💡 핵심 포인트**
- 에이전트가 읽는 모든 텍스트가 곧 공격 표면이 될 수 있음(프롬프트 인젝션 위험)
- 권한(scope) 최소화·입력 격리·출력 검열이 필수 방어책
- 조직 내 자동화·에이전트 도입 전 보안 설계(감사·로그·테스트)가 필요

**🤔 추가 인사이트 (한국 관점)**
한국 기업도 GitHub Actions 등 자동화 도구와 에이전트 결합을 빠르게 실험·도입하는 추세입니다. 따라서 에이전트가 접근할 수 있는 리소스 권한을 세분화하고, 외부 텍스트를 모델 입력으로 넣을 때는 시스템 레벨 지시(시스템 프롬프트)와 엄격히 분리하는 설계가 필요합니다. 보안팀·개발팀이 함께 공격 시나리오(레드팀)를 점검하고, 인시던트 대응 절차를 마련해야 합니다.

### 3. arXiv: DepthWeave-KV — 장문 컨텍스트용 KV 캐시 압축 기법 제안

> **원제:** DepthWeave-KV: Token-Adaptive Cross-Layer Residual Factorization for Long-Context KV Cache Compression
> **출처:** arXiv — https://arxiv.org/abs/2607.06523  · 접근일: 2026-07-08
> **발행:** 2026-07-08 (arXiv)

DepthWeave-KV는 LLM의 장문 컨텍스트 처리에서 키-값(KV) 캐시 메모리를 효율적으로 압축하는 새로운 방법론을 제안합니다. 저자들은 토큰-적응형(token-adaptive) 크로스-레이어 잔차 분해(residual factorization)를 통해 캐시 크기를 줄이면서도 복원 성능을 유지하거나 향상시키는 기술을 소개합니다. 실험에서 메모리 사용량과 추론 효율을 유의미하게 개선했다고 보고합니다.

장문 컨텍스트 운용은 대화형 에이전트·문서 요약·코드 이해 등에서 핵심적이므로, KV 캐시의 압축·전달 효율 개선은 실시간 응답성과 비용 모두에 직접적인 영향을 미칩니다. 다만 압축 과정에서의 정보 손실(예: 희귀 토큰 정보 희석)과 복원 시 지연-정확도 트레이드오프는 실제 시스템 적용 전에 면밀히 검증해야 합니다.

**💡 핵심 포인트**
- 장문 컨텍스트용 KV 캐시 압축으로 메모리·추론 비용 절감 가능
- 토큰 적응형 분해가 핵심 기여
- 상용 적용 전 희귀 정보 보존·복원 품질 검증 필요

**🤔 추가 인사이트**
한국 기업의 온프레미스·엣지 기반 LLM 서비스에서 KV 압축 기술은 인프라 비용을 크게 낮출 잠재력이 있습니다. R&D 팀은 압축-복원에 따른 응답 품질 변화를 산업별(금융·의료 등) 요구사항과 대조해 평가해야 합니다.

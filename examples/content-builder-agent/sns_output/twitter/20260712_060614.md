### 1. Grok Build CLI의 와이어-레벨 분석 — 사용자 코드·시크릿이 어떻게 전송·저장되는가

> **원제:** What xAI's Grok Build CLI Actually Sends to xAI
> **출처:** GitHub Gist / cereblab (https://gist.github.com/cereblab/dc9a40bc26120f4540e4e09b75ffb547)
> **발행:** 2026-07-10

xAI의 Grok Build CLI(consumer grok 0.2.93)에 대한 재현 가능한 와이어-레벨 분석입니다. 작성자는 mitmproxy와 캡처 로그를 이용해 CLI가 모델 턴(endpoint /v1/responses)뿐 아니라 별도의 저장 채널(/v1/storage)을 통해 작업 디렉터리 전체(git bundle 포함)를 Google Cloud Storage의 grok-code-session-traces 버킷으로 업로드하는 흐름을 포착했습니다. 특히 .env 같은 시크릿 파일이 모델 턴 바디에 원문으로 포함되는 사례와, 별도 채널로 저장 아카이브가 200 응답을 받아 수신되는 증거를 제시합니다.

내용은 기술적·증거 중심으로 서술되어 있으며, 핵심 발견은 (1) 읽은 파일의 내용이 모델 호출에 포함되어 전송됨, (2) 전체 리포지토리 스냅샷(또는 git bundle)이 별도의 업로드 채널을 통해 전송·수용됨, (3) 업로드 목적지로 GCS 버킷(grok-code-session-traces)이 식별된다는 점입니다. 작성자는 "Improve the model" 같은 토글을 꺼도 업로드가 계속되는 실험 결과를 보고합니다(설정이 전송/저장 동작을 차단하지 않음).

**💡 핵심 포인트**
- Grok CLI는 모델 턴과 별도로 작업공간 전체를 스냅샷하여 업로드(여러 75MB 청크, 200 응답)할 수 있음.
- `.env` 등 민감 정보가 모델 턴과 세션 아카이브에 원문으로 포함되어 전송되는 사례가 재현 가능하게 캡처됨.
- 업로드 대상은 Google Cloud Storage로 식별되며, 문서화·설정 안내에서 명확히 드러나지 않았다는 점이 문제로 지적됨.

**🤔 추가 인사이트**
기업·개발자 관점에서는 "로컬에서 실행하지만 데이터가 전송된다"는 사실을 배경으로 내부 정책·CI/CD 파이프라인·시크릿 관리 전략을 재검토해야 합니다. 특히 사내 코드나 비밀 키를 포함하는 저장소에서 Grok 같은 CLI를 실행할 때는 격리된 환경(예: 시크릿이 제거된 샌드박스, VM 스냅샷)이 필요합니다. 규제·컴플라이언스(의료·금융) 환경에서는 자동 업로드의 법적·계약적 영향을 따로 검토해야 합니다.

*출처: https://gist.github.com/cereblab/dc9a40bc26120f4540e4e09b75ffb547*

### 2. qMLX — Mac Studio에서 Qwen 3.5(122B)를 실사용 가능하게 만든 버그 수사와 최적화

> **원제:** qMLX: Maximising my AI psychosis by minmaxing my Mac Studio
> **출처:** mrzk.io (https://mrzk.io/posts/qmlx-maximising-ai-psychosis-minmaxing-mac-studio/)
> **발행:** 2026-07-10

한 개인(Andryo Marzuki)이 M3 Mac Studio Ultra에서 대형 모델(특히 Qwen 3.5 122B)을 실사용 가능한 수준으로 만들기 위해 겪은 문제와 해결 과정을 상세히 설명한 기술 블로그입니다. 핵심은 모델 자체의 성능 문제보다 서빙 스택(캐시·체크포인트·인터럽트 처리)에 숨어 있던 세 가지 버그를 찾아 고친 것이 모델을 '일상적인 페어 프로그래밍 도구'로 만든 결정적 요소라는 점입니다.

수정된 항목은 (1) 시스템 프롬프트에 매 턴 고유 ID를 찍어 캐시 불일치를 유발한 문제 제거, (2) 스트리밍 중 인터럽트 시 생성된 토큰을 영구 기록하지 않아 발생한 역사 누락 버그 보완, (3) 체크포인트 스토어에 생긴 쓰레기(매 256 토큰마다 생성된 비매칭 체크포인트)로 정당한 체크포인트가 강제 축출되는 문제의 회피 등입니다. 결과적으로 긴 문맥(수만~수십만 토큰)에서 이전과 달리 서브-초 응답을 재현할 수 있게 되었습니다.

**💡 핵심 포인트**
- 로컬 장비에서도 대형 모델을 실용화하려면 모델 외적(서빙·캐시·IO) 요소가 더 중요할 수 있음.
- 디스크 기반 KV 복원(캐시)과 적절한 체크포인트 관리로 긴 문맥 대화의 선형적 지연을 제거 가능.
- 동일 하드웨어(예: M3 Ultra)에서 모델 선택(Qwen3.5 122B)과 스택 개선(qMLX)이 결합돼 매우 응답성 높은 로컬 인퍼런스 환경을 달성.

**🤔 추가 인사이트**
한국의 연구·스타트업에서는 로컬 프론트엔드(데스크톱 워크스테이션)에 저지연 장기간 컨텍스트 에이전트를 배포할 때, 단순히 모델 용량만 고려하지 말고 캐시 정책·체크포인트 일관성·인터럽트 복구 같은 엔지니어링 문제를 먼저 점검하면 비용 대비 실사용성 개선 효과가 큽니다. 또한 M3 Ultra·고성능 애플 실리콘의 통합 메모리 특성에 맞춘 '활성 파라미터 크기' 선택(예: 122B)은 현실적 대안입니다.

*출처: https://mrzk.io/posts/qmlx-maximising-ai-psychosis-minmaxing-mac-studio/*

### 3. "100줄 Lisp"로 구현한 에이전트 — 호모아이코닉 언어가 재조명되는 이유

> **원제:** An Agent in 100 Lines of Lisp, or How my Prof was Right - Just 25 Years Early
> **출처:** The Beach (https://thebeach.dev/posts/lisp-agent/)
> **발행:** 2026-07-07

저자가 Common Lisp로 약 100줄 분량의 에이전트 루프를 구현하고, "코드가 곧 데이터"인 Lisp의 특성이 에이전트 설계에 자연스럽게 들어맞는 사례를 제시합니다. 핵심은 에이전트의 도구(툴) 카탈로그를 설계 시점이 아니라 런타임에 모델이 스스로 작성·평가(eval)해 확장할 수 있다는 점입니다. 예를 들어 모델이 자체적으로 HTTP 호출 함수를 생성해 웹 검색을 수행하는 등, 능동적으로 실행 가능한 코드를 생성·주입해 기능을 확장합니다.

실험은 eval을 단일 도구로 제공하는 방식으로 진행되며, 모델이 Lisp 폼을 생성해 로컬 이미지에 정의(defun)하고 즉시 호출하는 패턴을 보입니다. 이는 에이전트의 '기술(skill)'을 대화 내역(트랜스크립트)으로 저장하고 재수화(rehydrate)하는 방식으로, 런타임에 기능이 자가생성되는 새로운 패러다임을 시사합니다.

**💡 핵심 포인트**
- 에이전트 루프는 재귀 함수로 단순화 가능(메시지 리스트 → 모델 호출 → 도구 호출 시 재귀).
- Lisp의 호모아이코니시티(homoiconicity)가 에이전트가 스스로 도구를 생성·적용하도록 하는 데 강점.
- 실험적 접근(로컬 docker/sandbox 필요)은 강력하지만 eval 기반 실행은 보안 위험이 있어 프로덕션에서는 엄격한 격리가 필요.

**🤔 추가 인사이트**
한국의 연구·개발팀은 에이전트 플랫폼 설계 시 "런타임 확장성"을 고려해야 합니다. 기존 MCP/도구 등록 방식은 안전하고 관리 가능하지만, 모델이 스스로 코드를 생성해 기능을 확장하는 패턴은 향후 에이전트의 자율성·생산성을 높여 줄 수 있습니다. 다만 이 패턴은 실행 보안(권한·샌드박스)·검증(생성 코드의 정형성 증명) 문제가 병존하므로 엔터프라이즈 적용 전 안전 메커니즘을 설계해야 합니다.

*출처: https://thebeach.dev/posts/lisp-agent/*

### 1. sqlsure — AI가 생성한 SQL을 실행 전에 정형적으로 검사하는 도구 (GitHub)

> **출처:** GitHub (https://github.com/sqlsure/sqlsure)

sqlsure은 AI가 작성한 SQL 쿼리에서 흔히 발생하는 팬아웃(fan-out) 더블 카운팅, 불가산(additivity) 위반, 잘못된 조인키 등을 규칙 기반으로 판정해 실행 전에 오류를 잡아내는 정적 검사 도구입니다. dbt의 선언적 메타데이터를 규칙북으로 사용해 오프라인·결정론적으로 판정하며, 자동으로 수정(fix) 제안까지 제공합니다.

왜 주목할까?
- 에이전트가 데이터베이스에 직접 쿼리를 날리는 워크플로우에서 '검사-수정-실행' 파이프라인을 넣으면 실서비스 오류·데이터 유출·비즈니스 로직 오류를 크게 줄일 수 있습니다.

*링크:* https://github.com/sqlsure/sqlsure

Empero에서 공개한 Qwythos-9B는 Qwen3.5-9B를 기반으로 Claude 스타일의 긴 체인오브토트·툴 사용 데이터로 포스트트레이닝한 모델입니다. 특징으로 1,048,576 토큰(약 1M) 컨텍스트 지원, 네이티브 함수 호출(function calling), 도메인 특화(사이버보안·생물학·약리학) 성능 향상이 보고되어 있습니다. 라이센스는 Apache-2.0으로 공개되어 연구·실험에 활용 가능.

왜 주목할까?

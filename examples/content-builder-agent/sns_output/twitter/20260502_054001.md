### 1. LFM2가 24B MoE로 확장 — Liquid AI의 LFM2-24B-A2B 공개

> **원제:** LFM2-24B-A2B: Scaling Up the LFM2 Architecture
> **출처:** Liquid AI (https://www.liquid.ai/blog/lfm2-24b-a2b)
> **발행:** 2026-02-24

Liquid AI가 LFM2 계열의 상위 체크포인트인 LFM2-24B-A2B를 공개했습니다. 이 모델은 총 파라미터 240억(24B)이지만 토큰당 활성 파라미터는 약 23억(2.3B) 수준인 MoE(비활성 전문가 포함) 설계를 채택해 '총 용량은 크되 실제 추론 경로는 가벼운' 전략을 취합니다. 설계 목표는 32GB RAM 환경에서도 동작하도록 하여 클라우드뿐 아니라 일부 데스크탑/노트북·온디바이스 환경까지 배포성을 확보한 점이 특징입니다.

요약: LFM2-24B-A2B는 깊이(depth)와 전문가수(expert count)를 늘려 표현력을 확장하면서도, 토큰당 활성 경로를 작게 유지해 추론 비용·지연을 억제합니다. Liquid는 다양한 벤치마크에서 규모 대비 선형적인 품질 향상을 보고했으며 llama.cpp·vLLM·SGLang 등에서 즉시 실행 가능한 포맷(gguf 등)과 양자화 옵션을 제공합니다.

**💡 핵심 포인트**
- MoE로 총 파라미터는 크게 늘리되 활성 파라미터는 제한해 ‘엣지 친화적’ 배포 가능
- 여러 런타임(ltama.cpp, vLLM)과 양자화 포맷 지원으로 실험·배포 장벽 낮춤
- 벤치마크에서 규모 확장 시 품질 개선이 관찰되나, 실제 응용 성능·안정성은 실전 검증 필요

**🤔 추가 인사이트**
한국의 스타트업·기업 연구팀은 LFM2-24B-A2B 같은 오픈 웨이트·MoE 모델을 사용해 비용 대비 성능을 시험해볼 기회가 커졌습니다. 특히 온프레미스 배포·저지연 응용(기업 내부 검색, 도큐먼트 보조 등)에서 모델 선택의 폭이 넓어지므로, 인프라(메모리·NPU)와 양자화 파이프라인을 미리 준비해 두는 것이 유리합니다.

### 2. 일론 머스크 법정 증언 — xAI가 OpenAI 모델을 '부분적'로 활용해 Grok을 학습했다는 인정

> **원제:** Elon Musk testifies that xAI trained Grok on OpenAI models
> **출처:** Tim Fernholz / TechCrunch (https://techcrunch.com/2026/04/30/elon-musk-testifies-that-xai-trained-grok-on-openai-models/)
> **발행:** 2026-04-30

미국 연방법정에서 일론 머스크는 xAI가 OpenAI 모델(또는 공개 API/챗봇)을 대상으로 한 ‘distillation’(대규모 쿼리 기반 지식 추출) 기법을 사용해 Grok을 학습한 사례가 있다고 부분적으로 시인했습니다. Distillation은 방대한 질의응답을 통해 타 모델의 출력을 모방하거나 학습 데이터로 삼는 방식으로, 다른 사업자가 고성능 모델을 비교적 저비용으로 재현할 수 있게 하는 위험 요소로 논의되어 왔습니다.

기사에서는 이 행위가 명백히 불법이라기보다 서비스 약관·저작권·데이터 사용 정책 측면에서 문제의 소지가 있으며, OpenAI·Anthropic·Google 등은 모델 복제·추출을 차단하기 위한 기술·정책적 대응을 강화하고 있다고 전합니다.

**💡 핵심 포인트**
- 머스크의 증언은 ‘업계 내부에서도 distillation이 활용됐다’는 실무적 확인
- Distillation은 기술적으로 비용 효율적인 복제 수단이지만, 법적·정책적 위험 존재
- 대형 연구소들은 대량 쿼리 탐지·차단, API 사용 정책 강화 등으로 대응 중

**🤔 추가 인사이트**
한국 기업은 고객 데이터·자사 모델 보호 관점에서 API 사용 패턴 모니터링과 TOS 기반의 방어 전략(비정상 대량 쿼리 차단, 레이트리밋, 감사 로그 등)을 강화해야 합니다. 동시에 모델을 오픈하게 될 경우 데이터 유출·모델 탈취 리스크를 사전 고려한 계약·기술 조치를 마련할 필요가 있습니다.

### 3. 에이전트 오케스트레이션 시대의 재검토 — “In-Context Prompting Obsoletes Agent Orchestration” (arXiv)

> **원제:** In-Context Prompting Obsoletes Agent Orchestration for Procedural Tasks
> **출처:** Simon Dennis et al. / arXiv (https://arxiv.org/abs/2604.27891)
> **발행:** 2026-04-30 (arXiv 제출)

연구진은 전통적인 에이전트 오케스트레이션(외부 오케스트레이터가 상태를 추적하고 라우팅 지시를 반복적으로 주입하는 방식)이 절차적 업무(procedural tasks)에서는 더 이상 최선의 접근이 아닐 수 있음을 주장합니다. 동일한 모델을 사용하되 절차 전체를 시스템 프롬프트(또는 시나리오 형태의 컨텍스트)로 넣어 모델 자체에 자율적 오케스트레이션을 맡긴 실험에서, 외부 오케스트레이터보다 품질·성공률 면에서 우수한 결과를 얻었다고 보고합니다.

해당 논문은 여행 예약·원격 테크서포트·보험 클레임 등 세 도메인에서 비교 평가를 수행했고, in-context 방식이 실패율을 낮추고 평균 품질 점수에서 우위를 보였다고 결론짓습니다.

**💡 핵심 포인트**
- 고성능 LLM의 능력 향상으로 외부 오케스트레이터의 필요성이 줄어들 가능성 제시
- 절차가 명확하게 정의된 작업에서는 ‘절차 전체를 컨텍스트화’하는 방식이 강력
- 한계: 매우 장기·동적·외부상태 의존적 작업에서는 여전히 오케스트레이션이 유리할 수 있음

**🤔 추가 인사이트**
한국 기업의 에이전트 도입 전략은 ‘무조건 오케스트레이터 → agent’ 구조를 따르기보다, 먼저 모델에게 절차 전체를 주고 성능을 테스트해보는 A/B 실험을 권합니다. 단, 보안·감사 추적, 외부 시스템 호출 제어 등 운영상 요구사항은 여전히 외부 제어 레이어를 요구할 수 있습니다.

*출처(논문 PDF):* https://arxiv.org/pdf/2604.27891

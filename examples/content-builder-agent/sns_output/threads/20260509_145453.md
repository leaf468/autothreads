### 1. HTML을 ‘프롬프트 언어’로 활용한 Claude Code의 효과

> **원제:** Using Claude Code: The unreasonable effectiveness of HTML
> **출처:** Twitter / Hacker News 링크
> **링크:** https://twitter.com/trq212/status/2052809885763747935
> **발행:** 2026-05-09 (트윗/토론)

트윗과 Hacker News 토론을 통해 공유된 사례는, Claude Code(Anthropic의 코드/툴 중심 인터페이스)가 HTML 구조를 ‘제약된 출력 포맷’으로 사용했을 때 예측 가능하고 제어 가능한 코드·문서 출력을 얻는 데 매우 효과적이었다는 점을 강조합니다. 단순한 자연어 프롬프트 대신 HTML 태그·속성으로 모델의 출력 스킴을 강제로 규정하면, 파싱·후처리 비용이 줄고, 모델이 의도한 구조(테이블, 폼, JSON 유사 블록 등)를 안정적으로 생성하는 경향이 관찰됩니다.

이 접근은 특히 코드 생성·웹 UI 자동화·문서화 파이프라인에서 유용합니다. HTML은 사람이 읽을 수 있는 표현력과 동시에 구조화된 마크업을 제공하므로, 모델 출력의 '형식 강제'와 자동 검증(validator)에 적합합니다. 다만 트윗 토론에서는 보안(HTML 인젝션), 접근성, 그리고 HTML로 포맷할 때 발생할 수 있는 의미 손실(표현의 제약)에 대한 경고도 제시되었습니다.

**💡 핵심 포인트**
- HTML 태그를 프롬프트/출력 제약 언어로 사용하면 모델 출력의 구조적 일관성이 향상됨
- 파싱·검증 파이프라인과 결합하면 자동화 파이프라인의 신뢰성이 올라감
- 리스크: 인젝션·표현 제약·브라우저/환경별 렌더링 차이

**🤔 추가 인사이트 (한국 관점)**
국내 기업은 자동 문서화·웹 스크래핑·웹 기반 코드 생성 워크플로에서 HTML-기반 출력 규격을 채택하면 후처리 비용을 줄일 수 있습니다. 다만 HTML을 입력/출력 컨트롤 용도로 사용할 때는 XSS·렌더링 안전성 검사와, 의미 손실을 보완하는 보조 메타데이터(예: JSON-LD) 병행을 권장합니다.

### 2. Agentic AIs: 기초모델의 OOD(분포이탈) 문제를 푸는 ‘누락된 패러다임’

> **원제:** Agentic AIs Are the Missing Paradigm for Out-of-Distribution Generalization in Foundation Models
> **출처:** arXiv (Xin Wang et al.)
> **링크:** https://arxiv.org/abs/2605.06522
> **발행:** 2026-05-07 (arXiv 제출)

저자들은 기초모델(FM)이 실제 개방형 환경에서 마주하는 OOD 문제는 전통적 모델 중심(model-centric) 접근만으로는 구조적으로 한계가 있다고 주장합니다. 논문은 (1) 부분적으로 관찰되는 다단계 학습 분포를 허용하는 OOD의 형식화, (2) 파라미터 기반 표현의 커버리지 한계(파라미터 커버리지 천장) 증명, (3) 에이전트적(agentic) 시스템의 네 가지 구조적 속성(지각, 전략선택, 외부행동, 폐쇄루프 검증)을 제시하며, 이들이 모델 중심 접근을 넘어서는 해결책을 제공한다고 설명합니다.

핵심 주장 중 하나는 “어떤 입력들은 어떤 고정된 파라미터 집합으로도 허용오차 ε 내에서 처리할 수 없으며, 외부 세계와의 상호작용(액션)과 검증 루프를 포함한 에이전트적 구조만이 이를 확장할 수 있다”는 점입니다. 연구는 에이전트 방법이 모델 중심 방법을 대체하는 것이 아니라 보완하며, FM의 OOD 문제 해결을 위해 별도의 연구·벤치마크·인프라 투자가 필요함을 제안합니다.

**💡 핵심 포인트**
- OOD는 단순한 배포 shift 문제가 아니라 다단계·부분관찰 형식을 갖는 구조적 문제
- 파라미터 기반 접근에는 '커버리지 천장'이 존재함을 이론적으로 제시
- 에이전트적(외부 행동·검증 포함) 시스템이 해결 범위를 근본적으로 확장함

**🤔 추가 인사이트 (한국 관점)**
기업용 AI 도입 시 단일 모델의 성능만 믿기보다는, 탐색·실행·검증을 포함한 에이전트 아키텍처(예: 외부 검색, 툴 호출, 자동 검증 루프)를 설계하는 것이 장기적으로 더 견고합니다. 특히 금융·의료처럼 OOD 위험이 높은 도메인에서는 에이전트적 검증 파이프라인을 우선 도입해 리스크를 완화해야 합니다.

### 3. SFT(감독파인튜닝) 행동을 역전시키는 방법론 — 'Reversible SFT'

> **원제:** Crafting Reversible SFT Behaviors in Large Language Models
> **출처:** arXiv (Yuping Lin et al.)
> **링크:** https://arxiv.org/abs/2605.06632
> **발행:** 2026-05-07 (arXiv 제출)

이 논문은 감독 파인튜닝(SFT)으로 유도된 모델 행동을 기계적으로 '압축'해 기저의 희소 서브네트워크(carrier)로 만들고, 그 행동을 별도의 트리거(soft prompt)를 통해 역전(revert)할 수 있음을 보였습니다. 저자들이 제안한 LCDD(Loss-Constrained Dual Descent)는 라우팅 마스크와 가중치를 공동 최적화해 행동을 보존하는 희소 캐리어를 구성합니다. 그 위에서 SFT-Eraser라는 활성화 매칭 기반 soft prompt를 최적화하면 해당 SFT 행동을 손대지 않고도 강력하게 억제할 수 있습니다.

주요 결과는 ‘구조(희소한 채널/서브네트워크)’가 역전 가능성의 핵심이며, 단순 트리거 설계만으로는 동일한 효과를 얻을 수 없다는 점입니다. 이는 배포된 SFT 모델의 안전성·정책적 제어(특정 응답 차단 등)를 위한 새로운 도구로서 의미가 큽니다.

**💡 핵심 포인트**
- SFT로 학습된 행동을 인과적(연산적)으로 지역화 가능한 희소 캐리어로 압축 가능
- 캐리어 기반 설계와 soft-prompt 트리거를 결합하면 '비파괴적'으로 행동을 역전할 수 있음
- 운영 환경에서의 정책 적용(응답 차단·정책 롤백)에 유용

**🤔 추가 인사이트 (한국 관점)**
국내 서비스 운영자는 SFT 기반 모델을 배포할 때, 정책 변경·규정 준수 요구가 생겼을 경우를 대비해 '비파괴적 롤백'·'동적 억제 트리거' 설계(예: SFT-Eraser 유사 기법)를 검토하면 법적·컴플라이언스 리스크를 줄일 수 있습니다.

### 1. 구글, SpaceX에 월 9.2억 달러 규모의 컴퓨트 계약 체결 — 클라우드·에이전트 수요 급증 대응

> **원제:** Google will pay SpaceX $920M per month for compute
> **출처:** TechCrunch (Sean O'Kane)
> **URL:** https://techcrunch.com/2026/06/05/google-will-pay-spacex-920m-per-month-for-compute/
> **발행:** 2026-06-05

구글이 2026년 10월부터 2029년까지 매월 약 9.2억 달러를 지불해 SpaceX의 대규모 GPU·CPU 인프라(약 110,000대의 NVIDIA GPU 포함)를 확보하는 계약을 공개했습니다. TechCrunch 보도에 따르면 이는 최근 출시된 기업용 에이전트 제품(Gemini Enterprise 등)에 대한 예상보다 높은 수요를 메우기 위한 단기·중기 용량 확보 전략으로 보입니다. 계약엔 2026년 말 이후 90일 통지로 해지 가능한 조항이 포함되어 있어, 수요 변화에 따른 유연성도 담겼습니다.

기업 관점에서는 대형 클라우드 사업자들이 자체 인프라 외에 서드파티(특히 위성/데이터센터 사업자)의 컴퓨트 용량을 임대해 급증하는 AI 워크로드에 대응하는 ‘용량 확보 전략’이 점차 보편화되고 있음을 시사합니다. 한국의 클라우드·AI 벤더는 글로벌 공급망 변화와 함께 가격·가용성 리스크를 재평가해야 하며, 대형 계약의 해지·감액 리스크를 반영한 용량 계획이 필요합니다.

**💡 핵심 포인트**
- 대형 클라우드 사업자의 외부 대규모 컴퓨트 임대(월 수억 달러 규모)로 AI 인프라 구매·임대 시장이 확대
- 에이전트·생성 AI의 급격한 수요가 단기 용량 확보를 촉발
- 계약 유연성(해지 조항 등)이 수요 불확실성 대응의 핵심

### 2. Apple, Messages for Business에 첫 AI 에이전트 ‘Poke’ 승인 — 플랫폼 내 에이전트 상용화 가속

> **원제:** Apple approves Poke as the first AI agent on its Messages for Business platform (TechCrunch coverage)
> **출처:** TechCrunch (Apps / Sarah Perez 링크 요약)
> **URL:** https://techcrunch.com/2026/06/04/apple-approves-poke-as-the-first-ai-agent-on-its-messages-for-business-platform/
> **발행:** 2026-06-04

애플이 자사 Messages for Business 플랫폼에 최초의 AI 에이전트인 'Poke'를 승인함으로써, 플랫폼 차원에서 에이전트가 공식적으로 상용화될 길이 열렸습니다. TechCrunch 보도를 기준으로 Poke는 메시지 대화 흐름 속에서 상거래·고객지원·예약 처리 등을 자동화하며, 애플의 앱 생태계와 메시징 인터페이스를 통해 즉시 접근 가능하다는 점이 특징입니다. 애플 승인은 개인정보·프라이버시 기준을 엄격히 적용하는 환경에서 에이전트가 플랫폼 인증을 받았다는 의미도 큽니다.

한국 시장에서는 iMessage/비즈니스 메시징 채널을 통한 고객 경험 자동화가 활성화될 수 있으며, 특히 모바일 퍼스트 서비스(배송·예약·쇼핑 등)는 플랫폼 인증 에이전트를 통해 사용성·신뢰성을 동시에 얻을 가능성이 큽니다. 다만 애플의 개인정보·심사 기준을 만족시키려면 에이전트의 데이터 최소화·로컬 처리·명확한 사용자 동의 체계가 필수입니다.

**💡 핵심 포인트**
- 플랫폼 차원의 에이전트 승인으로 상용 배포·신뢰도 장벽이 낮아짐
- 모바일 메시징을 통한 에이전트 접점이 고객 지원·커머스에 빠르게 확산될 가능성
- 개인정보·심사 기준 준수가 제품 설계의 전제 조건

### 3. arXiv: "Learning What to Forget" — LLM의 ‘언러닝(unlearning)’을 토큰 수준에서 개선하는 연구

> **원제:** Learning What to Forget: Improving LLM Unlearning via Learned Token-Level Importance
> **출처:** arXiv (Gizem Yüce et al.)
> **URL:** https://arxiv.org/abs/2606.06320
> **발행:** 2026-06-04 (arXiv)

본 논문은 LLM에서 특정 지식(데이터)을 제거하는 ‘머신 언러닝’ 문제에 있어, 기존의 샘플 단위 접근 대신 토큰 단위의 중요도를 학습해 더 효과적으로 잊게 만드는 방법(ATWU)을 제시합니다. 핵심 아이디어는 어떤 토큰이 '잊혀져야' 하는지 판단할 때 보존(retain) 목표와의 충돌(conflict)을 측정 기준으로 삼는 것이며, 간단한 선형 스코어러를 통해 토큰별 가중치를 학습함으로써 잊기-보존 트레이드오프를 개선합니다.

실험(TOFU, RWKU 벤치마크)에서 ATWU는 기존 방법들보다 성능이 우수했으며, 학습된 토큰 가중치가 의미론적으로도 잊혀야 할 영역과 잘 정렬되는 결과를 보였습니다. 실제 운영 환경(컴플라이언스 요청·데이터 삭제 등)에서 '선택적 언러닝'을 효율적으로 구현하는 데 실용적 기여가 기대됩니다.

**💡 핵심 포인트**
- 토큰 수준 중요도 학습으로 언러닝의 정밀도 및 효율성 개선
- 보존 목표와의 충돌을 기준으로 한 정식화가 실무적 적용성 높음
- 규제·컴플라이언스 대응(데이터 삭제 요구) 솔루션으로 유망

### 1. Microsoft / pg_durable — PostgreSQL 내부의 내구성 있는 실행(워크플로) 확장

> **출처:** GitHub (microsoft/pg_durable)
> **URL:** https://github.com/microsoft/pg_durable

pg_durable는 PostgreSQL 내부에서 체크포인팅 기반의 내구성 실행(durable execution)을 제공하는 확장입니다. SQL 수준에서 워크플로를 정의하고 각 단계별로 상태를 영속화해 데이터베이스 재시작·충돌 상황에서도 작업을 재개할 수 있게 합니다. 벡터 임베딩·RAG 파이프라인, 대량 인게스트 작업, 데이터 전처리 파이프라인 등 DB 근처에서 신뢰성 있게 파이프라인을 운영하고자 하는 팀에 유용합니다.

왜 주목할까?
- AI 파이프라인(임베딩·업서트·스케줄링)을 DB 인접에서 신뢰성 있게 운영하면 인프라 복잡도와 장애 표면을 줄일 수 있음
- 온프레미스·규정 준수 환경에서 외부 오케스트레이터 의존도를 낮출 수 있음

### 2. Agent-Reach — 에이전트의 ‘인터넷 접근성’ 설치·통합 툴

> **출처:** GitHub (Panniantong/Agent-Reach 트렌딩)
> **URL:** https://github.com/Panniantong/Agent-Reach

Agent-Reach는 에이전트가 트위터(X), 유튜브, 레딧, 깃허브, Bilibili, 샤오홍슈 등 다양한 온라인 채널을 '명령 한 줄'로 접근할 수 있게 해주는 오픈 소스 설치기·스크립트 모음입니다. Cookie 기반 접근·yt-dlp·rdt-cli 등 기존 도구들을 조합해 에이전트가 외부 컨텐츠를 읽고 요약하도록 돕습니다.

왜 주목할까?
- 많은 에이전트 활용 사례가 외부 웹 데이터 접근을 전제로 하므로, 접근성 툴의 성숙은 실사용성(Research, RAG, 모니터링)을 크게 높임
- 국내 환경에서 API 비용·접근 제약을 우회해 연구·PoC를 빠르게 진행할 수 있음(단, 서비스 약관·보안·프라이버시 규정 준수 필요)

### 3. PaddleOCR 3.6 — 문서·테이블·차트까지 겨냥한 경량 VLM 중심의 문서 파싱 업그레이드

> **출처:** GitHub (PaddlePaddle/PaddleOCR) — Release v3.6.0
> **URL:** https://github.com/PaddlePaddle/PaddleOCR

PaddleOCR 3.6은 문서용 비전-언어 모델(PaddleOCR-VL-1.6)과 구조화 파이프라인(PP-StructureV3)을 중심으로 문서 파싱·표·수식·고문서 인식 능력을 크게 향상시켰습니다. 결과물을 Markdown/JSON 포맷으로 바로 내보내는 기능이 강화되어 RAG·문서형 LLM 애플리케이션의 전처리 파이프라인에 바로 연결하기 편리합니다.

왜 주목할까?
- LLM 기반 문서 이해/검색 애플리케이션에서 고품질 구조화 데이터(표 좌표·셀 단위 정보 등)를 얻는 것이 전처리의 핵심인 만큼 실용성 높음
- 한국어·다국어 문서 처리 지원이 강화되어 국내 문서 RAG 프로젝트에 즉시 적용 가능
